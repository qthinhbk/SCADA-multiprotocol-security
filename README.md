# SCADA-multiprotocol-security

Nền tảng mô phỏng bảo mật SCADA đa giao thức, dùng để xây dựng và kiểm thử toàn bộ chuỗi phòng thủ công nghiệp:

**Sinh traffic -> Phát hiện xâm nhập -> Phản ứng tự động -> Cưỡng chế chặn -> Quan sát bằng dashboard**

Project hỗ trợ 4 giao thức phổ biến trong môi trường OT/ICS:

- Modbus TCP
- DNP3
- IEC 60870-5-104
- OPC-UA

---

## 1. Project này giải quyết bài toán gì?

Trong mạng SCADA thật, một hệ thống chỉ “monitor” là chưa đủ. Hệ thống cần:

1. Phát hiện tấn công theo ngữ cảnh giao thức công nghiệp.
2. Chuyển cảnh báo thành hành động chặn tự động.
3. Bảo vệ luồng điều khiển quan trọng khi có flood/congestion.
4. Có số liệu vận hành để quan sát và đánh giá hiệu quả.

Project này mô phỏng đầy đủ các yêu cầu trên trong môi trường Docker.

---

## 2. Tổng quan kiến trúc

### 2.1 Thành phần chính

- **SCADA services**: server/client/attacker cho từng giao thức.
- **Zeek IDS**: phân tích lưu lượng và sinh cảnh báo (`notice.log`).
- **ids-monitor**: đọc cảnh báo, phân loại threat, cập nhật blacklist.
- **active_blocker**: đọc `blacklist.acl` và tạo rule `iptables LOG + DROP`.
- **firewall-monitor**: điểm cưỡng chế chính trên `FORWARD` datapath.
- **qos-router**: áp QoS (HTB) để ưu tiên traffic hợp lệ.
- **InfluxDB + Grafana**: lưu trữ và trực quan hóa metric.

### 2.2 Luồng xử lý bảo mật

`Attacker traffic -> Zeek alert -> ids-monitor -> blacklist.acl -> active_blocker -> iptables ACTIVE_BLOCK -> packet bị DROP`

### 2.3 Các chế độ triển khai

#### A) Mô hình cơ sở (không bảo mật cưỡng chế)
- Compose file: `docker-compose.yaml`
- Mục đích: chạy lab cơ bản, quan sát traffic và dịch vụ.

#### B) Mô hình có bảo mật kiểu co-located
- Compose file: `docker-compose.secure.yaml`
- Mục đích: tương thích/kiểm thử nhanh.

#### C) Mô hình routed inline enforcement (khuyến nghị)
- Compose file: `docker-compose.inline.yaml`
- Mục đích: cưỡng chế đường đi qua firewall thực sự.
- Đây là chế độ đầy đủ nhất để kiểm thử IDS/IPS + QoS.

---

## 3. Kiến trúc mạng của chế độ inline

Ở chế độ inline, mỗi giao thức dùng 2 subnet riêng:

- **Source subnet**: client + attacker
- **Destination subnet**: server

Firewall là gateway (`*.254`) giữa hai subnet, nên traffic liên subnet **bắt buộc** đi qua `FORWARD` chain.

`zeek-ids` và `qos-router` chia sẻ network namespace với `firewall-monitor` để cùng quan sát/xử lý đúng datapath.

> Lưu ý: Zeek đang chạy near-real-time theo cơ chế pcap xoay vòng (không phải live inline DPI 100% từng packet).

## 4. Cài đặt nhanh

### 4.1 Yêu cầu hệ thống

- Docker Desktop (hoặc Docker Engine + Compose plugin)
- Windows PowerShell (nếu chạy script `.ps1`)
- RAM khuyến nghị từ 8GB trở lên

### 4.2 Clone và chạy

```powershell
git clone <repo-url>
cd SCADA-multiprotocol-security
```

Chạy mô hình inline đầy đủ:

```powershell
docker compose -f docker-compose.inline.yaml up -d --build
```

---

## 5. Cách triển khai

### 5.1 Giai đoạn không bảo mật (mô hình cơ sở)

Chạy toàn bộ stack cơ sở:

```powershell
docker compose -f docker-compose.yaml up -d --build
```

Kiểm tra nhanh:

```powershell
docker ps --format "table {{.Names}}\t{{.Status}}"
docker logs --tail 50 modbus-server
docker logs --tail 50 iec104-server
docker logs --tail 50 dnp3-server
docker logs --tail 50 opcua-server
```

### 5.2 Giai đoạn bảo mật (inline mode - khuyến nghị)

Project có script tự động demo:

```powershell
.\inline-demo.ps1 -Step reset
.\inline-demo.ps1 -Step start
.\inline-demo.ps1 -Step attack -Protocol modbus
.\inline-demo.ps1 -Step verify
```

Chạy toàn bộ giao thức:

```powershell
.\inline-demo.ps1 -Step full -Protocol all
```

Thêm kịch bản flood để kiểm thử QoS:

```powershell
.\inline-demo.ps1 -Step full -Protocol all -WithFlood
```

## 7. Reset môi trường trước mỗi lần test

```powershell
Set-Content .\zeek_scripts\blacklist.acl "# Runtime blacklist entries generated by ids_monitor.py`n# Reset before each demo run to avoid historical noise.`n"
Remove-Item .\zeek_pcaps\*.pcap -ErrorAction SilentlyContinue
Remove-Item .\zeek_pcaps\processed\* -ErrorAction SilentlyContinue
Remove-Item .\zeek_pcaps\failed\* -ErrorAction SilentlyContinue

docker compose -f docker-compose.inline.yaml up -d --build --force-recreate firewall-monitor zeek-ids ids-monitor
```

## 8. Cấu trúc thư mục quan trọng

- `docker-compose.yaml`: mô hình cơ sở
- `docker-compose.secure.yaml`: mô hình có bảo mật co-located
- `docker-compose.inline.yaml`: mô hình bảo mật routed inline
- `inline-demo.ps1`: script chạy demo tự động
- `firewall/`: firewall monitor, rules, active blocker
- `zeek_scripts/`: Zeek rules, ids-monitor, blacklist
- `qos/`: logic QoS và phân lớp ưu tiên
- `grafana/`: provisioning dashboard

---

## 9. Giới hạn hiện tại

- Chế độ `secure` vẫn là co-located, không cưỡng chế datapath như inline.
- Zeek phân tích theo batch pcap xoay vòng (near-real-time).
- DNP3 trong lab có phần mô phỏng qua TCP/20000 để phục vụ test nhiều kịch bản.
